# Product

## Register

product

## Users

LLM fine-tuning learners and hands-on developers using a local Mac or ordinary development machine. They want to understand data preparation, tokenization, Trainer loops, LoRA, PEFT, SFT, DPO, QLoRA planning, evaluation, checkpoints, adapters, and practical hardware boundaries without starting from large-scale training.

They use the interface while studying and running local experiments. The primary job is to configure a small training run, inspect every step, compare base versus adapter output, and connect the Studio workflow back to the 01-10 course sequence.

## Product Purpose

LLM Study Studio turns a local fine-tuning study repo into a usable training and learning workspace. Success means a learner can see where data enters, how tensors and labels change, what model state is being updated, what artifact was produced, and which course explains the concept.

The product should make small experiments feel trustworthy and legible. It should emphasize local-first constraints, reproducible paths, visible commands, and clear separation between independent Studio runs and course traces.

## Brand Personality

Precise, technical, and calm. The interface should feel like a serious local lab bench for model training, not a decorative AI dashboard.

The tone should be direct and Chinese-friendly, with English retained where it is the real technical vocabulary: SFT, LoRA, PEFT, DPO, QLoRA, JSONL, adapter, trace.

## Anti-references

Avoid generic neon AI dashboards, purple-blue gradient-heavy SaaS screens, glassmorphism, toy-like cards, marketing hero layouts, and noisy repeated tiles. Avoid burying the training command, data schema, run state, or adapter path behind decorative panels.

Avoid the current over-dark, over-glowing, visually crowded page feel. The interface should be denser and more organized, but not cramped.

## Design Principles

1. Make the training path visible: method, model, data, parameters, run, compare, artifacts.
2. Treat local constraints as product information: model size, quick mode, device, generated paths, and hardware limits should be visible at the moment they matter.
3. Keep course learning connected to practice: every Studio concept should have a nearby course path, and every course run should expose trace, metrics, model state, and Chat Lab context.
4. Prefer instrument-panel clarity over decoration: hierarchy, alignment, state, and data density matter more than visual effects.
5. Preserve trust in local execution: commands, file paths, process status, logs, and adapter availability should be explicit.

## Accessibility & Inclusion

Target WCAG AA contrast for text and controls. Support keyboard navigation, visible focus states, reduced motion, non-color-only status communication, readable Chinese text, and responsive layouts that do not clip labels or controls on narrow screens.
